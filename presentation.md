# ESP32 Coin Detection Presentation Script

Ниже текст для презентации к защите. Он ориентирован на выступление примерно на 10 минут.

Важно: часть истории разработки ниже восстановлена по текущему репозиторию, README и финальным артефактам. Если какие-то шаги на ПК делались немного иначе, перед защитой просто подправь формулировки под фактический процесс.

## Общая Логика Выступления

Главная идея выступления:

1. Показать, что задача была реальной и ограниченной по ресурсам.
2. Показать понятный pipeline от данных до прошивки.
3. Подчеркнуть, что проект не остановился на offline-метриках, а реально был доведён до ESP32-CAM.
4. Отдельно показать инженерные проблемы и то, как они решались.

## Слайд 2. Архитектура Решения

Что вывести на слайд:

`Сбор данных -> Подготовка датасета -> Обучение baseline -> Оценка -> int8 квантизация -> Экспорт firmware bundle -> Запуск на ESP32-CAM -> HTTP demo`

Что сказать:

"Pipeline был таким: сначала данные размечались через Edge Impulse, затем локально готовился датасет в удобный JSON-формат. После этого обучалась компактная FOMO-style модель, потом она оценивалась на held-out test, затем экспортировалась в int8 TFLite, превращалась в C++ headers и встраивалась в firmware ESP32-CAM. На устройстве модель запускалась через TensorFlow Lite Micro, а результат отдавался через HTTP endpoints."

Что подчеркнуть:

"Edge Impulse использовался только для разметки. Обучение, оценка, квантизация и deployment были локальными."

Что сказать:

"Всего в модели 47 617 параметров. После int8-квантизации deploy-модель занимает примерно 59.8 KB, что уже выглядит разумно для микроконтроллера."

## Слайд 8. Инженерные Проблемы И Как Они Решались

Что вывести на слайд:

Проблема 1:
`Mismatch между train/eval pipeline и firmware preprocessing`

Проблема 2:
`Нестабильность при разном освещении`

Что сказать:

"В проекте было две интересные инженерные проблемы. Первая проблема состояла в том, что модель сама по себе была не самой слабой, но реальный firmware preprocessing не совпадал с pipeline на ПК. Из-за этого на устройстве модель могла вести себя хуже, чем в offline-eval."

"Эта проблема решалась выравниванием pipeline: основной путь был переведён на JPEG/QVGA -> RGB888 -> resize -> quantize -> infer. То есть preprocessing на устройстве стал ближе к preprocessing при обучении и оценке."

"Вторая проблема возникла уже позже: при изменении освещения модель снова начинала недосчитывать монеты. Для этого был сделан отдельный lighting-robust retrain с усиленными фотометрическими аугментациями: exposure, gamma, saturation, hue, color scaling, random shadows и noise. Кроме того, firmware postprocessing был выровнен с offline-eval."

Ключевая мысль:

"То есть качество улучшалось не одной магической заменой модели, а последовательным инженерным выравниванием всего pipeline."

## Чего Лучше Не Делать На Защите

Не стоит:

1. Долго показывать train-логи по эпохам.
2. Фокусироваться только на `bin_acc`.
3. Скрывать ограничения по скорости.
4. Говорить, что "модель просто стала лучше" без объяснения почему именно.

Лучше говорить так:

"Главный результат был достигнут за счёт инженерной настройки всего pipeline: представления данных, декодирования heatmap, preprocessing на устройстве и аугментаций под реальные условия съёмки."
